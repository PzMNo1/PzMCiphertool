• 建议先别从“面板好看不好看”入手，而是先定 API 中转站到底要管什么真实对象。现在最 值
  得先做的是这条主线：

  第一阶段：真实密钥管理
  先把“API 密钥”做成真实后端能力。原因是它是 API 中转站的根对象，余额、用量、模型 分
  发、请求日志都应该挂在某个用户、某个 key、某个模型调用下面。

  建议目标：

  - 用户登录后可以创建 API Key。
  - 后端生成真实 key，只在创建时完整返回一次。
  - 后端只保存 key hash，不保存明文。
  - 前端列表只显示 mask，例如 sk-pzm-****-9f21。
  - key 有状态：启用、暂停、删除。
  - key 有额度字段：总额度、已用额度、剩余额度。
  - 所有数据按登录邮箱隔离。

  第二阶段：真实请求代理
  然后做真正的“中转站”。也就是前端或外部调用：

  POST /v1/chat/completions
  Authorization: Bearer sk-pzm-...

  后端收到后：

  - 验证 API Key 是否存在、启用、未超额。
  - 根据模型路由配置，转发到真实上游模型供应商。
  - 记录请求日志。
  - 统计 token、耗时、花费。
  - 返回上游响应。

  这一步完成后，面板里的“今日请求、今日消费、Token、平均响应”才有真实来源。

  第三阶段：用量日志与统计聚合
  等请求代理跑通，再做统计层：

  - 每次调用写一条 usage log。
  - 字段至少包括：用户邮箱、keyId、model、requestTime、latency、inputTokens、
    outputTokens、cost、status。
  - 面板数据从 usage log 聚合，而不是手写数字。
  - 支持按 24h、7 天、30 天查询。
  - 模型分布、趋势图、最近使用都从同一批日志聚合。

  第四阶段：余额/额度系统
  这个可以先简化，不急着接支付。

  先做：

  - 每个用户一个账户余额。
  - 每个 key 可以设置 quota。
  - 每次请求后扣费。
  - 余额不足或 key 超额就拒绝请求。
  - 兑换码/充值入口后面再接。

  第五阶段：路由规则
  最后再做模型路由：

  - 用户选择默认上游供应商。
  - 模型别名映射：例如 gpt-5-mini -> 某个真实 provider model。
  - 限流策略：RPM、TPM、每日预算。
  - fallback：主供应商失败时切备用供应商。

  我建议当前下一步先做 真实 API Key 管理，不要直接做用量面板。因为没有真实 key，就没
  有真实调用入口；没有调用入口，所有统计还是只能模拟。

  最小可行版本可以这样定：

  1. 后端新增 ApiKey 数据结构。
  2. 支持创建、列表、启用/暂停。
  3. 创建时返回一次明文 key。
  4. 后端保存 hash、mask、quota、status、createdAt。
  5. 前端 API 中转站的“创建 API 密钥”和表格接这个真实接口。


  完成这个以后，再接 /v1/chat/completions 代理接口，面板数据就能自然变真。

      - POST /v1/chat/completions 已能用你的 sk-pzm-* key 调用。

  3. 第三阶段：真实用量统计
      - 每次调用写入 usage 日志。
      - 面板从真实日志聚合：
          - 请求数
          - Token
          - 最近使用
          - 模型分布
          - 平均响应
          - key 已用量
      - 已确认账号隔离正常：哪个账号创建的 key，usage 就归哪个账号。
  4. 第四阶段第一版：配额和限流
      - 新建 key 默认配额改成 1,000,000 tokens。
      - 后端调用前检查：
          - key 是否存在
          - 是否暂停
          - 是否超 Token 配额
          - 是否超过 RPM
          - 是否超过 TPM
      - 限流/超额会返回 OpenAI 兼容错误：
          - 429 rate_limit_exceeded
          - 429 insufficient_quota
      - 配置项已放进 application.yml，可用 .env 覆盖：
          - API_ROUTER_RPM_LIMIT
          - API_ROUTER_TPM_LIMIT
          - API_ROUTER_DEFAULT_TOKEN_QUOTA
          - API_ROUTER_INPUT_PRICE_PER_MILLION
          - API_ROUTER_OUTPUT_PRICE_PER_MILLION
